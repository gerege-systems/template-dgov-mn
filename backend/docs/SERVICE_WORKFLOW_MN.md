# Төрийн үйлчилгээний бүртгэл ба хүсэлтийн урсгал

Энэ баримт нь үйлчилгээ хэрхэн бүртгэгддэг, иргэний хүсэлт хэрхэн шийдвэрлэгддэгийг
тайлбарлана. Хамрах хүрээ: migration 44 (хүсэлтийн workflow), 45 (регистр),
47 (нэгтгэл), 48 (эрхийн цэвэрлэгээ).

## 1. Нэг эх сурвалж: регистр

Үйлчилгээний **мастер өгөгдөл нь `registry_services`** (CPSV-AP паспорт).
`gov_services` нь түүний **ажлын проекц** — иргэний портал, хүсэлтийн workflow
энэ хүснэгт дээр ажиллана.

```
registry_services  ──(нийтлэх)──▶  gov_services  ◀── gov_applications
   (паспорт, мастер)                 (проекц)         (иргэний хүсэлт)
        │
        └── registry_service_evidences ──▶ иргэнээс шаардах баримтын жагсаалт
```

**Дүрэм:** `gov_services`-ийг ГАРААР засахгүй. Паспортоо засаад дахин нийтэлнэ —
`ProjectToGov` нь ажлын каталогийг паспортын агуулгатай тэнцүүлнэ. Холбоос нь
`gov_services.registry_service_id` (UNIQUE).

| Үйлдэл | Үр дагавар |
|---|---|
| Паспорт нийтлэх | `gov_services` мөр үүсэх/шинэчлэгдэх, `enabled=true` |
| Паспорт архивлах | `enabled=false`, `lifecycle='withdrawn'` (мөр УСТАХГҮЙ) |
| Нотолгоог `from_citizen=false` болгох | Иргэнд харагдах баримтын жагсаалт богиносно |

Архивлахад мөрийг устгадаггүйн шалтгаан: `gov_applications.service_id` нь түүн
рүү заасан байдаг тул устгавал иргэний хүсэлтийн түүх тасарна.

## 2. Олон улсын кодчилол

| Талбар | Стандарт | Жишээ |
|---|---|---|
| `code` | ISO 3166 + COFOG суурьтай | `MN-0133-002` |
| `cofog_code` | НҮБ COFOG 1999 | `01.3.3` |
| `main_activity` | ЕХ main-activity authority table | `gen-pub` |
| `sdg_code` | SDG (EU 2018/1724) Annex II procedure | `S1` |
| `processing_time` | ISO 8601 duration (`cv:processingTime`) | `P7D` |
| `output_type` | CPSV-AP Output толь (`cpsv:produces`) | `Declaration` |
| `assurance_level` | eIDAS (Reg. 910/2014 Art.8) | `substantial` |
| `registry_life_events.eu_code` | ЕХ life/business event толь | `RES`, `STBU` |

**Анхаар:** CPSV-AP-ийн "COFOG main activity Authority Table" нь ЖИНХЭНЭ COFOG
БИШ — тэр нь ЕХ-ны худалдан авалтын main-activity жагсаалт. Тиймээс НҮБ-ын
COFOG-ийг `cofog_code`-д тусад нь хадгалдаг.

SDG Annex II-ийн `R1`–`X6` кодууд нь зохицуулалтын эх текстэд байхгүй, SEMIC-ийн
codelist давхарга — тиймээс зөвхөн ҮНЭН таарсан үед л бөглөнө.

## 3. Хоёр урсгал: auto ба manual

`registry_services.fulfilment` нь иргэн юу хүлээхийг тодорхойлно.

### `auto` — шууд биелнэ

Бүртгэлээс уншиж олгодог лавлагаа/тодорхойлолт. Хүн оролцохгүй: хүсэлт, лавлагаа,
мэдэгдэл **нэг транзакцид** үүсч, төлөв шууд `completed` болно. Менежерийн
дараалалд ОРОХГҮЙ.

### `manual` — менежер шийдвэрлэнэ

Хүсэлт `registered` төлөвтэй үүсч, SLA хугацаа тамгалагдаж дараалалд орно.
Иргэнд "хүлээн авлаа" мэдэгдэл өгнө.

Энэ хуваалт нь EU 2018/1724 Art. 6(2)-той нийцнэ: гаралт шууд олгогдохгүй бол
хүлээн авсан тухай автомат мэдэгдэл (b), гүйцэтгэл дууссан тухай мэдэгдэл (d)
өгөх ёстой; шууд олгогдож байвал эхнийх нь шаардлагагүй.

### Автоматжуулалтын эрх зүйн шалгуур

`fulfilment='auto'` нь `has_discretion` ба `has_assessment` **хоёулаа false**
үед л хадгалагдана. Энэ нь Германы VwVfG §35a-ийн загвар: шийдвэрийг бүрэн
автоматаар гаргах нь (1) хуулиар зөвшөөрөгдсөн, (2) үнэлэх эрх (Ermessen)
байхгүй, (3) урьдчилсан нөхцөлд үнэлгээний зай (Beurteilungsspielraum) байхгүй
үед л боломжтой. Usecase давхарга үүнийг албадана; давхар хамгаалалт болгож
`applyAuto` нь тэмдэглэгээтэй үйлчилгээг гараар хянуулах руу шилжүүлдэг.

## 4. Төлөвийн машин

```
submitted ──▶ registered ──▶ in_review ──▶ approved ──▶ completed
                  │              │  ▲          
                  │              ▼  │          
                  │        info_required       (цаг ЗОГСОНО)
                  │              │
                  └──────────────┴──▶ rejected / cancelled / expired
```

Зөвшөөрөгдсөн шилжилтүүд нь `domain.govTransitions`-д тодорхойлогдсон бөгөөд
`domain.GovCanTransition`-оор шалгагдана. Repository давхарга нь мөн адил
дүрмийг SQL `WHERE status IN (…)` guard болгон ДАХИН хэрэгжүүлнэ — тэр нь
уралдааныг (хоёр менежер зэрэг "Батлах" дарах) хаах давхарга; 0 мөр хөндөгдвөл
`apperror.Conflict`.

**`approved` vs `completed`:** гаралт нь лавлагаа бол шийдвэр гармагц олгогдож
`completed` болно. Гаралт нь БИЕТ зүйл (үнэмлэх, гэрчилгээ) бол `approved`-д
хүлээж, хэвлэгдэж хүргэгдсэний дараа officer `complete` үйлдлээр хаана.

**Үр дүнгийн толь** (`result`) нь төлөвөөс тусдаа: `granted`, `refused`,
`withdrawn`, `not_admissible`, `processed`. Голландын ZGW-ийн зарчим — явцын
толийг байгууллага өөрөө тодорхойлдог ч ҮР ДҮНГИЙН толийг нэгтгэдэг.

## 5. SLA ба цаг зогсоох

`due_at` нь хүсэлт үүсэх үед **нэг удаа** тамгалагдана (уншилт бүрт дахин
тооцвол хугацаа "гулсаж" зөрчлийг нуух байсан).

`info_required` төлөвт шилжихэд `suspended_at` тамгалагдаж **SLA цаг зогсоно**.
Иргэн мэдээллээ өгөхөд `due_at` нь зогссон хугацааны туршид хойшилно. Ингэснээр
иргэний удаашрал байгууллагын зөрчил болж бүртгэгдэхгүй (Голландын Awb 4:15-ийн
загвар).

Background worker (`SLASweep`, 60с тутам) хоёр зүйл хийнэ:

1. **Хугацаа хэтрэлт** — `sla_breached` латч тавьж иргэнд нэг удаа мэдэгдэнэ.
   Цаг зогссон хүсэлтийг АЛГАСНА.
2. **Чимээгүй зөвшөөрөл** — `tacit_approval=true` үйлчилгээний хугацаа хэтэрсэн
   хүсэлтийг зөвшөөрөгдсөнд тооцож `tacit=true` тэмдэглэнэ. Шийдвэр автоматаар
   гарсныг иргэнд ил мэдэгдэнэ.

> Чимээгүй зөвшөөрлийн ойлголт нь Европын зохицуулалтад (Services Directive
> 2006/123/EC Art. 13(4)) тулгуурласан боловч **Монголд шууд хүчин төгөлдөр
> биш**. Үйлчилгээ тус бүрийн эрх зүйн үндэслэлийг Монголын хууль тогтоомжоор
> баталгаажуулах ёстой; өгөгдмөл нь `false`.

`SLASweep` нь HTTP хүсэлт биш тул context-д identity байхгүй — RLS нь
identity-гүй үед бүх мөрийг хаадаг. Тиймээс sweep нь `rls.WithService(ctx)`-ээр
системийн үүрэг заавал тавина.

## 6. Эрх ба RLS

| Эрх | Хэн | Юу |
|---|---|---|
| `gov.review` | manager (id=3) | Иргэний хүсэлт хянаж шийдвэрлэх |
| `registry.view` / `registry.manage` | admin | Паспорт, нотолгоо, нийтлэлт |

**`officer` RLS үүрэг.** Менежер бусад иргэний хүсэлтийг харах ёстой тул
`service|admin|user` толинд багтахгүй. `OfficerRLSContext` middleware нь ЗӨВХӨН
`/gov/officer/*` route-уудад `app.user_role='officer'` тавина.

Least-privilege: `gov_applications`, `gov_references`, `gov_notifications`,
`gov_application_events`-д л officer бодлого өгсөн. `users`, `gov_payments`,
`gov_appointments`-д бодлого БАЙХГҮЙ — RLS permissive (OR) тул officer тэнд тэг
мөр харна (fail-closed). Өөрөөр хэлбэл менежер иргэний төлбөр, цаг захиалга,
бүртгэлийн мэдээлэлд хүрэхгүй.

Middleware-ийг глобалаар суулгаж БОЛОХГҮЙ — тэгвэл менежер өөрийн профайлаа ч
харахаа болино.

## 7. Ул мөр

Хүсэлтийн шилжилт бүр `gov_application_events`-д append-only бичигдэнэ (хэн,
хэзээ, ямар төлвөөс ямар төлөв рүү, ямар үндэслэлээр). Иргэн өөрийн хүсэлтийн
түүхийг УНШИНА (`WITH CHECK (false)` — бичихгүй); менежер бүгдийг харна.

Татгалзах шийдвэр нь **үндэслэлгүй байж болохгүй** — usecase давхарга хоосон
тайлбартай татгалзлыг татгалзана. Иргэн юунд татгалзсаныг мэдэж, гомдол гаргах
эрхтэй.

## 8. Шинэ үйлчилгээ нэмэх

1. `/admin/registry/services` дээр паспорт үүсгэнэ (ноорог).
2. Нотолгоо холбоно. ХУР-д байгаа баримтыг `from_citizen=false` болгоно — эс
   тэгвээс once-only зөрчил болж, зарласан проактив байдлын шатанд нийтлэх
   боломжгүй.
3. Үйл ажиллагааны тохиргоо: `fulfilment`, `sla_hours`, `output_type`,
   `output_ref_type` (лавлагаа гаргадаг бол), `assurance_level`.
4. Нийтэлнэ → `gov_services` руу автоматаар буух ба иргэн хүсэлт гаргаж чадна.

Ажлын каталогт ГАРААР мөр нэмэх шаардлагагүй бөгөөд нэмэх ёсгүй.
